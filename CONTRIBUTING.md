# Contributing to LumaFetch

Thank you for considering contributing to LumaFetch! This document provides guidelines and instructions for contributing.

## Code of Conduct

Please be respectful and constructive in your interactions. We welcome contributors of all backgrounds and experience levels.

## How to Contribute

### Reporting Bugs

Before creating bug reports, please check existing issues as you might find out that you don't need to create one. When you are creating a bug report, please include as many details as possible:

* Use a clear and descriptive title
* Describe the exact steps to reproduce the problem
* Provide specific examples to demonstrate the steps
* Describe the behavior you observed after following the steps
* Explain which behavior you expected to see instead and why
* Include screenshots if possible
* Include details about your environment (OS version, app version, etc.)

### Suggesting Enhancements

Enhancement suggestions are tracked as GitHub issues. When creating an enhancement suggestion, please include:

* Use a clear and descriptive title
* Provide a detailed description of the suggested enhancement
* Explain why this enhancement would be useful
* List some examples of how this enhancement would be used

### Pull Requests

* Fill in the required template
* Follow the project's coding style
* Include appropriate tests if applicable
* Update documentation as needed
* Ensure your code passes all existing tests

## Development Setup

### Prerequisites

* Node.js 18 or later
* Python 3.9 or later
* Git

### Installing Dependencies

```bash
# Install Node.js dependencies
npm install

# Install Python dependencies
pip install -r requirements.txt
```

### Running the Application

```bash
npm start
```

### Building the Application

```bash
npm run build
```

## Coding Style

### JavaScript/Node.js

* Follow ESLint rules (to be added)
* Use meaningful variable names
* Add comments for complex logic
* Keep functions small and focused

### Python

* Follow PEP 8 style guide
* Use type hints where appropriate
* Add docstrings for public functions and classes
* Keep functions small and focused

## Testing

Please ensure your changes don't break existing functionality. Tests will be added in future versions.

## Documentation

* Update README.md if you change functionality
* Update CHANGELOG.md with your changes
* Add inline comments for complex code sections

## Questions?

Feel free to open an issue for any questions about contributing.
